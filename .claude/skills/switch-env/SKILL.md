# Switch Env

Toggle between local Docker Supabase (development) and production Supabase. No manual `.env.local` editing needed.

## How it works

- `.env.local` — production values (gitignored, always loaded by Next.js)
- `.env.development.local` — local values (gitignored, only loaded in `npm run dev`, overrides `.env.local`)

When `.env.development.local` exists, Next.js loads it with highest priority in dev mode. When it doesn't exist, `.env.local` (production) is used.

This skill writes or removes `.env.development.local`.

## Switch to local

```
User: switch to local
User: use local db
User: /switch-env local
```

### Step 1: Check Docker

```bash
docker info &> /dev/null
```

If Docker is not running, tell the user to start Docker Desktop first. Stop here.

### Step 2: Check Supabase CLI

```bash
supabase --version &> /dev/null
```

If not installed: `npm install -g supabase`. Stop here.

### Step 3: Start local Supabase

```bash
supabase status &> /dev/null || supabase start
```

`supabase start` may take a minute on first run (pulls Docker images). If it fails, print the error and stop.

### Step 4: Get local keys

```bash
supabase status 2>&1
```

Parse the output for:
- `anon key:` — the value after this label
- `service_role key:` — the value after this label

If parsing fails, show the raw output and ask the user to provide the keys.

### Step 5: Write `.env.development.local`

Create the file at the project root:

```
# Local Supabase (Docker) — auto-generated by switch-env skill
# This file overrides .env.local in dev mode (Next.js priority)
# Delete this file to switch back to production
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_OR_ANON_KEY=<parsed anon key>
SUPABASE_SERVICE_ROLE_KEY=<parsed service_role key>
```

### Step 6: Confirm

Tell the user: "Local env active. Run `npm run dev`."

If `.env.local` doesn't exist, warn the user they'll need it when switching back to production.

## Switch to production

```
User: switch to production
User: use production db
User: /switch-env production
```

### Step 1: Remove `.env.development.local`

```bash
rm -f .env.development.local
```

### Step 2: Verify `.env.local` exists

```bash
test -f .env.local && echo "exists" || echo "missing"
```

If `.env.local` is missing, warn: "No `.env.local` found. Production env vars are not configured. Copy from `.env.example` or ask a teammate."

### Step 3: Confirm

Tell the user: "Production env active (`.env.local`). Run `npm run dev`."

Also mention they can run `supabase stop` to free Docker resources.

## Edge cases

### Supabase not linked / no config.toml

If `supabase/config.toml` is missing, `supabase start` will fail. Tell the user to run `supabase init` first.

### Local Supabase already running

`supabase start` is idempotent. If already running, it just confirms. If `supabase status` shows it's running, skip `supabase start`.

### supabase status output format changed

If the key labels in `supabase status` output don't match `anon key:` or `service_role key:`, show the user the raw output and ask them to provide the values manually.

### RESEND_API_KEY for local dev

Email is handled by Inbucket locally (http://127.0.0.1:54324). No Resend key needed for local dev. Don't include RESEND_API_KEY in `.env.development.local` — it'll inherit from `.env.local` if needed.
